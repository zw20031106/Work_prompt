# 标准战不是技术评测 | Deck Brief

**项目定位。** 这是一套 12 页以内的管理层叙事型 deck，目标不是复述历史八卦，而是用三场经典标准战把“市场真正为谁投票”讲成一个今天依然可用的决策框架。

**目标读者。** 目标读者是企业战略负责人、产品/平台负责人、投资人、研究人员，以及需要在技术路线和生态路线之间做判断的管理层。

**主使用场景。** 使用场景是战略讨论、研究分享、课堂演示和 board-style 沟通。读者应在 15 分钟内完成一次认知纠偏：标准不会奖励孤立的技术最优，标准奖励最容易被生态一起采用的方案。

**目标动作。** 读者在看完后应能用同一套五因素框架去判断新的标准战、平台战、协议战和生态战，而不是继续把规格表当作主胜负手。

**内容边界。** 本 deck 只使用 `demo_draft/设计方案.md` 已给出的论点和来源方向，不额外扩展成百科式历史综述。页面重点是机制、对比、联盟、切换成本和 adoption 逻辑。

**交付物要求。** 最终交付应包含 `brief.md`、`deck_narrative.md`、派生 `build/generated/slide_specs.yaml`、可编辑 `pptx`、逐页预览图、至少 1 页 connector 结构校验结果、视觉复核日志和 handoff 说明。

**验证要求。** 必须使用 `cail` conda 环境完成构建。必须导出预览图并逐页人工复核。必须对 connector 页执行 XML 级别结构校验。缺少任何一项都不算完成。

**风格目标。** 风格参考 editorial + boardroom 的混合语言：米白浅底、深墨文字、冷暖双强调色、强标题句、稳定网格、结论优先。标题使用中文判断句，保留必要的英文案例名与术语。

**版式目标。** 每页只保留一个主 archetype。优先使用 process flow、comparison matrix、research note 和 decision logic。避免大段 bullet、弱标题和历史流水账式时间轴。

**素材与字体约束。** 页面以原生文本、形状、原生表格、原生 Office chart、必要的 Python figure 和 connector 为主，尽量保持可编辑性与证据表达能力的平衡。标题优先使用 `Noto Serif CJK SC`，正文优先使用 `Noto Sans CJK SC`。

**不允许发生的错误。** 不允许把 source、页码或说明文字挤进标题区。 不允许为了塞内容把正文字号压到 `14pt` 以下。 不允许把历史案例做成“谁规格更高谁更惨”的口号页而缺少机制解释。 不允许静默降级或只交付脚本不交付预览证据。
