# Slide Design System

**这份文档的定位。** 本文把页面原型和共享视觉规则放在一起，目标是让 agent 在设计一页时，同时知道“这页该长成哪种 archetype”和“这页至少要满足哪些视觉底线”。它是 `design_support.md` 之下的页面级细则文档。

## 什么时候先读它

**当你已经知道要做哪些页，但还没决定每页怎么读、怎么排、怎么分层时，先读这份文档。** 它是 deck 规划完成后最重要的页面级文档。

## Archetype 总表

| Archetype | 主要任务 | 阅读方式 | 典型场景 |
| --- | --- | --- | --- |
| `hero-statement` | 定调 / 开场 | `scan` | 开场页、章节页、核心判断页 |
| `decision-logic` | 说服 / 拍板 | `decision` | go / no-go、管理层判断 |
| `board-memo` | 摘要 / 管理层同步 | `decision` | 路线图摘要、管理层 brief |
| `research-note` | 解释 / 研究表达 | `guided` | 技术说明、研究结论、数据机制 |
| `war-room-board` | 现场感 / 主链路 | `guided` | incident、应急、运营指挥板 |
| `process-flow` | 顺序解释 | `guided` | 用户流程、审批流、系统流程 |
| `chart-spotlight` | 证据 / 读图结论 | `decision` | KPI、趋势、证据页 |
| `comparison-matrix` | 比较 / 取舍 | `reference` | 方案比选、before/after |
| `appendix-dense` | 存档 / 查阅 | `reference` | 附录、备查页、详细数据 |

## Archetype 使用原则

**一页只选一个主 archetype。** 图、数、文可以混用，但不能让一页同时承担两个主 archetype。

**开场页优先考虑 `hero-statement`、`decision-logic`、`board-memo`。** 区别在于你要的是一句判断、一个决策树，还是一页管理层摘要。

**解释页优先考虑 `research-note`、`war-room-board`、`process-flow`。** 区别在于你更强调层级关系、现场感，还是顺序流程。

**证据页优先考虑 `chart-spotlight`。** 当页面核心是“让图支撑一个结论”时，不要把它伪装成 dashboard。

## Archetype 误用

**不要让开场页偷偷变 appendix。** 一页如果同时想给结论和塞细节，往往两边都失败。

**不要把 `chart-spotlight` 做成 dashboard。** 一页如果有太多图却没有主结论，它就不再是 spotlight。

**不要把 `comparison-matrix` 做成 bullet 列表。** 如果任务是比较，就应让比较关系显式可读。

## 标题区

**标题区属于高价值信息。** 左上角优先留给标题、subtitle、eyebrow、结论句和页面角色定义。

**弱信息必须系统降权。** 页码、图号、source、note、confidential、logo 等内容应进入页脚、边角或右下，不得抢占标题区。

**标题最好像答案。** 高质量 deck 的标题更适合直接表达“这页想让读者记住什么”，而不是只给一个主题名词。

## 网格与留白

**所有主元素都要挂到公共网格。** 标题、卡片、图表、注释和页脚都应沿稳定边线或基线对齐。

**同类元素要共享对齐轴。** 文本块左线齐，卡片顶部齐，数字列右线齐，图表组沿统一边界展开。

**父容器里的重复子项应尽量均匀分布。** 当一个 shape 或 panel 内承载多个小卡片、小标签、小步骤块或案例条目时，这些子项应优先按稳定间距平均排列，而不是局部挤在一起、另一侧留出大块空白。

**留白是结构。** 留白用来分组、隔离和强调，不是“空出来所以好看”。

**左右边距应先统一。** deck 级页面应先定义一组稳定的 `left_margin / right_margin`，标题区、主内容区和底部统计区都围绕同一版心展开，不要让左边更紧、右边更松。

## Typography

**有参考模板时先读真实文字与字号。** 在开始写新页之前，先确认封面标题、正式页标题、正文、图注、页脚和页码分别使用什么字号区间。字号关系本身就是模板身份的一部分。

**中文正文默认小四约 `12pt`。** 在没有更强模板或品牌约束时，中文正文、卡片正文、解释句、步骤说明和普通注释默认使用小四约 `12pt`，英文 fallback 使用 Times New Roman。英文 deck、现代产品 deck 或参考模板已形成更大字号系统时，可以通过 style profile 使用 `14pt` 正文主档位。

**模板字号优先于通用默认。** 只要参考模板已经有稳定的字号梯度，就优先沿用模板自己的层级。中文正文小四约 `12pt` 是回退值，不是覆盖模板的统一命令；英文或现代商务 deck 可以在 style profile 中使用更大的正文主档位。

**字号层级应收敛成少数几档。** 推荐至少统一为 `title`、`subtitle`、`body`、`label`、`badge/caption` 这几类 token，不要每个模块各写一套字号。

**离散字号梯度优先。** 默认优先使用 `12 / 14 / 16 / 18 / 20 / 24` 这类稳定梯度，而不是在同一套 deck 里到处出现 `13`、`15`、`17`、`13.5` 这种碎片字号。

**正文主档位按语言 profile 固定。** 中文任务默认以小四约 `12pt` 作为最广泛使用的正文档位；现代英文或产品风格 deck 可以以 `14pt` 作为正文主档位。比正文更小的弱层级文字可以存在，但不应反过来占主导。

**一级标题默认 `24pt`。** 正式内容页的一级标题默认使用 `24pt`。高于正文的其他层级优先只从 `16 / 18 / 20` 中选择。

**标题类文本默认使用 `1.0` 倍行距。** 一级标题、章节标题、副标题、panel header 和其他承担阅读锚点职责的标题类文本，默认使用 `1.0` 倍行距。标题不是正文段落，优先目标是建立节奏与层级，而不是填满行间空白。

**标题类文本默认保留 `0.5` 行的段前与段后空间。** 这部分空间用来隔离标题块和周围元素，让标题像一个清晰的视觉单元，而不是贴着图表或正文硬挤在一起。

**正文类文本默认使用 `1.5` 倍行距。** 正文、卡片正文、解释句、步骤说明、图旁说明和大部分连续阅读文本，默认使用 `1.5` 倍行距。正文的目标是让人持续阅读，而不是为了省空间去压缩行距。

**中文正文默认使用正式段落规则。** 中文任务的正文段落默认首行缩进 2 个中文字符，段前段后各 `0.5` 行，`1.5` 倍行距。标题类文本仍按标题规则处理，表格内文本按 table policy 处理，不把正文首行缩进带进表格单元格。

**默认 token 表应尽量少而完整。** 没有更强模板约束时，建议优先按下面这套字号 token 组织整套 deck。

| Token | 中文默认 pt / 值 | 典型用途 |
| --- | --- | --- |
| `hero_title_font_pt` | 24 | 封面、章节开场、hero statement |
| `section_title_font_pt` | 20 | 章节页、过渡页、大卡片主标题 |
| `page_title_font_pt` | 24 | 正式内容页一级标题 |
| `subtitle_font_pt` | 16-18 | 副标题、重点解释句、次级标题 |
| `minor_title_font_pt` | 14-16 | panel header、卡片小标题、局部强调标题 |
| `body_font_pt` | 12 | 正文、解释句、普通卡片文本 |
| `label_font_pt` | 10.5-12 | 标签、badge、chart axis、data label、diagram node label |
| `caption_font_pt` | 9-10.5 | 图注、source、footer、page tag、页码 |
| `body_first_line_indent_chars` | 2 | 中文正文首行缩进 |
| `body_paragraph_space_lines` | 0.5 | 中文正文段前 / 段后行数 |
| `title_line_spacing_multiple` | 1.0 | 标题类文本默认行距 |
| `body_line_spacing_multiple` | 1.5 | 正文类文本默认行距 |
| `title_paragraph_space_lines` | 0.5 | 标题类文本默认段前 / 段后行数 |

**相同语义应复用同一 token。** 不要因为对象类型不同就私自新增字号档位。`chart axis`、`legend`、`badge`、`page tag` 和 `diagram node label` 这些弱层级对象，应优先挂到 `label` 或 `caption`，而不是再造一批 `10pt`、`11pt`、`12.5pt` 的局部例外。

**标题和标签可以更大，但正文不要低于 profile 主档位。** 为了适配长标题，可以压缩标题文案或重排标题区；不要默认通过把中文正文降到小于小四、英文 / 现代商务正文降到小于既定 profile 主档位来换空间。

**窄容器应先重构，再谈字号。** 当某个卡片或流程条容器无法承载当前正文主档位时，优先改容器宽度、网格或文案密度，而不是回到不可读的小字。

**中英混排应显式定义字体策略。** 在没有品牌模板要求时，中文任务的普通正文、卡片正文、图注和表格文本默认采用中文宋体、英文 Times New Roman。标题和标签可以在此基础上做风格化选择，但不应让 Office chart、表格和普通文本使用彼此割裂的字体系统。

**sans-serif 风格应成对切换。** 如果任务明确需要更现代的产品、运营或品牌展示气质，可把 style profile 切换为中文黑体、英文 Arial，并在 chart、table、caption 与正文中同步应用。

## 色彩与几何

**颜色先定义角色。** 推荐先定义 `background`、`surface`、`text`、`accent`、`semantic`、`data series` 的职责。

**高亮只服务少数对象。** 一页里真正高亮的对象应该很少，通常是一个结论、一组关键数字或一条主路径。

**默认可以采用 `85 / 10 / 5`。** 大约 `85%` 中性色与背景、`10%` 主强调色、`5%` 语义色或特殊高亮。

**角语言必须统一。** 同一套 deck 里的主要卡片、节点和信息框，应保持一致的角语言，不要混合直角与圆角而没有语义解释。

## 图表与图片

**图表先服务结论。** 图表类型应响应页面 message，而不是响应作者手边最熟的函数。

**会后要改数时优先原生 Office chart。** 这类图更适合后续继续编辑。

**高信息密度图优先高 DPI 图片。** 热力图、复杂排序图、时间条图更适合图片化，但插入时必须保比例。

**不要强行拉伸图片。** 宁可出现可控留白，也不要把图像比例拉坏。

## 表格与矩阵

**真正的数据表优先原生表格。** 当页面本质是在展示结构化数据、行列关系、明细表头或附录数据时，优先使用 Office 原生表格，不要用 PPT shape 人工拼出“看起来像表格”的结构。

**中文表格默认五号与单倍行距。** 中文任务的原生表格默认使用五号约 `10.5pt`，单倍行距，段前段后 `0`，无特殊缩进。图表下方的数据表也按这套规则执行，不因为尺寸较小就临时降成不可读的小字。

**表格单元格上下默认居中。** 所有表格单元格默认 `vertical_anchor = middle`，避免表头、index 和数值贴到上边界或下边界。行高不足时优先调行高、表格高度或列宽，再处理字号。

**表格水平对齐按语义决定。** 表头 / columns 标签居中；index / 类目列居左；普通文本列居左；财报、经营指标和百分比等数值列靠右。对齐规则应由列语义或显式 `numeric_columns` 决定，不应靠视觉猜测。

**表格语义要进入复核。** final 前 visual review 应抽查表格是否按列语义执行对齐：表头是否居中，index / 类目列是否居左，文本列是否居左，数值列是否右对齐，所有单元格是否上下居中。

**`comparison-matrix` 保留表达自由度。** 比较页既可以使用原生表格，也可以使用矩阵卡片、评分格或对比条形图。关键标准是比较关系是否显式可读，而不是机械追求所有矩阵都长成表格。

## 正式研报 / 财报点评

**研报型 deck 在中文默认 typography 之上增加视觉纪律。** 这类 deck 应启用 `domain_profile: financial_report_review` 或同类 profile，可以使用深红、暗金、浅米灰等低饱和体系，保持稳定页眉、标题区、主版心、页脚、来源注和免责声明位置。不要使用任何具体机构的 logo、受保护标识或可能造成背书误解的品牌元素。

**图表页应有完整读图结构。** 正式研报 / 财报点评页默认包含图号、发现式图题、单位说明、来源注和必要的数据表。图表可以使用原生 Office chart 保持可编辑性，也可以使用高 DPI Python figure 承载复杂研究图，但字体、标题、单位和表格对齐仍应服从同一 profile。

**版心稳定比装饰更重要。** 表格、图表和摘要框应挂到同一版心，右侧点评框、底部数据表和页脚应跨页复用同一边界。视觉增强应服务阅读路径，不应让页面变成白底默认图表加零散卡片。

## 复核底线

**以下问题视为 `fatal`。** 弱信息占标题区、文本与背景对比严重不足、图表明显拉伸、装饰层压正文、一页没有单一视觉中心。

**以下问题视为 `warning`。** 网格不齐、卡片间距不稳、标题只有主题没有结论、图例离数据太远、同类页面风格漂移。

**以下问题视为 `preference`。** 标题还可更像 answer title、色彩预算还能更收、同 archetype 的视觉变体还能更统一。
