# Build Routes

**这份文档的定位。** 本文定义 `word-polished-doc-collab` 在不同文档复杂度下应选择什么技术路线，避免把所有任务都粗暴塞进同一种构建方式。

## Route A: `docx -> markdown -> docx`

**这是当前最适合合同、制度、说明文档和批量整理任务的主路线。** 适用场景：
- 收到的是外部 `.docx`
- 后续会持续改稿
- 需要统一风格
- 结构以标题、正文、列表、表格、图片为主

**优点是 source 和 delivery 分离。** 原件保留，Markdown 易维护，最终 `.docx` 统一导出，最适合版本可追溯的工作方式。

## Route B: `markdown -> docx`

**这是新写文档的高效率路线。** 适用场景：
- 原始内容本来就在 Markdown
- 主要工作是组织结构和统一版式
- 图、表、图片都能通过相对路径接入

**前提是 Markdown 语义必须完整。** 表题、图题和表注不能省略，否则最终 `.docx` 无法稳定排版。

## Route C: `markdown -> docx -> ooxml patch`

**这是精细格式控制路线。** 适用场景：
- 需要精确处理 Word 样式槽位
- 需要字段、复杂编号、分节、页眉页脚
- 需要 Office 原生对象或更细的兼容性修补

**这条路线更重，但边界更清楚。** 它适合高要求正式文件，不适合被默认滥用。

## Route D: `python figure -> assets -> markdown -> docx`

**这是复杂图表的主路线。** 适用场景：
- 研究图、热力图、密度图、机制图
- Office 原生 chart 难以稳定表达
- 图表先以图像资产交付即可

**图表仍然是文档对象。** Python figure 只是资产生成方式，不改变文档的标题、图题、表注和 style profile 规则。

## Route E: `office native chart / illustration in docx`

**这是精细模式的重要可编辑资产路线。** 适用场景：
- 收件人需要在 Word 里继续编辑图表
- 收件人需要在 Word 里继续编辑 KPI 卡片、简单流程框、标签组或说明型插图
- 数据标签、系列颜色、图例和轴格式需要继续维护
- 文档是模板化业务材料，而不是一次性静态报告

**这条路线应显式建设，不做静默降级。** 如果宿主系统还没有原生 visual 能力，就应明确说“当前只有图片路线”，而不是把图片冒充原生对象。

## 选路原则

**先按业务边界选路。** 文本统一、复杂图表、模板复用和 Office 原生对象是不同问题，不该混成一个脚本分支。

**先选最简单且足够正确的路线。** 普通费用文档和合同体系文档，优先走 `docx -> markdown -> docx` 即可，没有必要一开始就进复杂 OOXML patch。

**资产模式应成体系选择。** Office 原生 visual 负责可编辑图表和插图，Python figure 负责高复杂度研究图。两条路线并列存在，不应互相伪装。

**需要更强格式控制时，再进入更重路线。** 当字体槽位、字段或原生 visual 已经成为交付要求时，再切换到更精细的实现，不要提前复杂化。
